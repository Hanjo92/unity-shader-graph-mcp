# 1.0.0 Work Split

This document turns the final `1.0.0` checklist into concrete parallel work packages.

The goal is to ship final `1.0.0` without reopening broad engine scope. The package-backed editing engine is already strong enough. The remaining work is mostly transport, installability, boundary hardening, and release packaging.

## 1.0.0 Outcome

Final `1.0.0` should mean:

- the existing package-backed Shader Graph engine stays stable
- a real live MCP transport path is available
- a fresh checkout can be installed and smoke-tested
- the supported runtime boundary is explicit in docs

## Critical Path

These must land before `1.0.0`:

1. live MCP transport binding
2. end-to-end external client smoke test
3. fresh-project install and package import smoke test
4. release docs and boundary docs frozen to actual shipped behavior

## Parallel Streams

## Stream A: Server Transport

Write scope:

- `server/src/unity_shader_graph_mcp/`
- `server/tests/`

Owns:

- final MCP transport binding
- tool registration through the transport layer
- request/response path from live client to existing tool handlers
- Python tests for live transport behavior

Deliverables:

- real transport adapter replacing the current transport-agnostic gap
- stable CLI and MCP entrypoints
- tests proving tool listing and one end-to-end request path

Must not edit:

- `packages/unity-shader-graph-mcp/`
- `docs/`
- `contracts/`

Exit criteria:

- an MCP client can call the shader graph tool through the server
- one happy-path graph operation works through the live transport path

## Stream B: Unity Hardening

Write scope:

- `packages/unity-shader-graph-mcp/Editor/`
- `packages/unity-shader-graph-mcp/Tests/Editor/`

Owns:

- Unity-side validation and error message cleanup
- final smoke coverage for the current supported matrix
- install-time and fresh-project reliability fixes inside the package

Deliverables:

- clearer unsupported-node and unsupported-port messages
- EditMode smoke tests that lock the release-critical paths
- one fresh blank-graph happy-path test kept stable

Must not edit:

- `server/`
- `docs/`
- `contracts/`

Exit criteria:

- current EditMode suite stays green
- the release-critical happy path is stable in a fresh Unity project

## Stream C: Contracts And Docs

Write scope:

- `contracts/`
- `docs/`

Owns:

- supported boundary language
- install instructions
- end-to-end happy-path documentation
- release notes and operator guidance

Deliverables:

- frozen `1.0.0` runtime boundary doc
- install guide for Unity package plus server startup
- one recommended happy-path flow from graph creation to save
- release notes aligned to the real shipped behavior

Must not edit:

- `packages/unity-shader-graph-mcp/`
- `server/`

Exit criteria:

- a new user can follow the docs from fresh checkout to first successful graph mutation
- the docs do not promise unsupported node or port coverage

## Main Agent

Owns:

- repository root files
- integration review across streams
- final release checklist run
- final version/tag/changelog coordination

Should do after Streams A-C land:

- run final smoke pass
- confirm `CHANGELOG.md`, release body, and `1.0.0` checklist align
- cut the final release tag

## Recommended Execution Order

1. Start Stream A and Stream C in parallel.
2. Start Stream B once the `1.0.0` supported boundary language is frozen enough to know what must be tested.
3. Let the main agent do the final fresh-project smoke pass after all three streams settle.

## Suggested First Sub-Agent Round

If we split immediately, use this first round:

- Agent 1: implement live MCP transport in `server/src/` and `server/tests/`
- Agent 2: tighten Unity-side error messages and release-critical EditMode smoke tests in `packages/.../Editor` and `packages/.../Tests/Editor`
- Agent 3: write install docs, happy-path docs, and freeze `1.0.0` boundary language in `docs/` and `contracts/`

## Explicit Non-Goals For 1.0.0

- all nodes
- all ports
- template-backed graph creation beyond blank graphs
- broad texture, UV, or custom-function expansion

Those belong after `1.0.0`, most likely in `1.1.x`.
